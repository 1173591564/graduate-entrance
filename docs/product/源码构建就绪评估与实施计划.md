# 11408 备考系统：源码构建就绪评估与实施计划

## 结论

**可以开始构建源码，但应按“可运行纵切片”分阶段推进，不建议一次性铺开整个系统。**

当前材料已足够支撑：

- 后端、Web、Android 的工程骨架与统一 API 契约；
- 考纲导入、阶段配置、可用时段、确定性排程；
- 今日任务、计时、打卡、顺延、日历；
- 后续题库、SM-2、AI、RAG、代码验证、看板的扩展接口。

在正式编码前需要完成一轮小型“设计冻结”，重点不是继续补页面，而是把数据边界、同步语义和异步任务模型定下来。以下事项可与工程初始化同步完成，不需要等待全部内容都完善后才开工。

## GitHub 与部署方案补充

### 仓库现状

部署说明给出了既定 GitHub 仓库。通过用户工作机执行只读 `git ls-remote` 后确认：

- 远程仓库可访问；
- 当前没有任何 branch、tag 或 HEAD，属于空仓库；
- 当前工作区尚未执行 Git 初始化；
- Devin 的 GitHub 集成目前看不到该仓库，因此暂时不能使用标准 PR/CI 管理能力。
- 将仓库设为 public 只解决匿名读取，不会自动授予 Devin 创建分支、推送和管理 PR 的写权限。

开工前建议完成两个一次性动作：

1. 用户在 GitHub 空仓库中通过“Add a README”建立 `main` 基线分支；
2. 将该仓库授权给 Devin/Cognition GitHub App，使其出现在 Devin 的可访问仓库列表。

完成后，所有源码变更从 `devin/...` 功能分支进入，通过 PR 合并到 `main`，不直接向 `main` 推送。

### 工作区布局调整

既然 GitHub 仓库就是整个 `graduate-entrance` 项目，建议把当前工作区根目录作为仓库根，而不是再嵌套一层同名仓库：

```text
graduate entrance/
├── docs/
│   ├── 11408备考系统设计.md
│   └── ui-reference/
├── seed/
│   └── syllabus/
├── backend/
├── web/
├── android/
├── infra/
└── docker-compose.yml
```

初始化时再安全地移动现有设计、参考图和考纲；迁移前先校验文件数量与哈希，避免丢失。

### 凭据安全

当前部署说明文件包含服务器高权限登录凭据，**绝不能提交到 GitHub**。正式初始化仓库前应：

- 把该文件加入 `.gitignore`；
- 创建不含账号、密码和密钥的 `docs/deployment.md`；
- 立即轮换当前服务器密码；
- 新建专用 `deployer` 用户，使用 SSH key，禁止远程 root 密码登录；
- 应用密钥放入服务器 `.env` 或 GitHub Actions Secrets；
- GitHub 中至少配置 `DEPLOY_HOST`、`DEPLOY_USER`、`DEPLOY_SSH_KEY`，不保存明文 root 密码。

### 推荐部署拓扑

生产服务器继续采用设计中的阿里云 ECS：

```text
GitHub PR
  → CI: backend/web/android checks
  → main
  → build versioned images/artifacts
  → SSH deployer
  → docker compose pull/up
  → health check
  → failed health check rolls back previous image tag
```

服务器侧：

- Caddy 只开放 80/443；
- SSH 端口限制来源 IP，或至少启用 fail2ban；
- PostgreSQL、code_runner、worker 不映射公网端口；
- Docker Compose 使用固定镜像版本，不使用无界 `latest`；
- 数据库、图片和备份使用持久化目录；
- 部署前自动执行数据库备份；
- Alembic 迁移失败时停止发布，不继续启动新版本；
- `/api/health/live` 与 `/api/health/ready` 分开；
- 第一次正式上线前做一次完整恢复演练。

### CI 分层

建议 GitHub Actions 分为：

1. `ci-backend`：ruff、类型检查、单元测试、迁移检查；
2. `ci-web`：lint、typecheck、build；
3. `ci-android`：lint、unit test、assembleDebug；
4. `integration`：PostgreSQL + API 集成测试；
5. `deploy-production`：仅 `main` 且全部检查通过后执行，生产环境需人工批准。

## 开工前必须修正的设计点

### 1. 考纲层级与数据模型不一致

CSV 实际是：

`科目 → 模块 → 章 → 节 → 知识点`

现有模型只有：

`Subject → Module → KnowledgePoint`

首版迁移前应补 `Chapter`、`Section`，或者改为统一的自引用 `SyllabusNode`。建议采用显式层级表，便于类型约束、排序和前端树形展示：

```text
Subject
Module(subject_id)
Chapter(module_id)
Section(chapter_id, nullable)
KnowledgePoint(chapter_id, section_id nullable)
```

政治没有“节”，允许 `section_id` 为空。

### 2. 考纲缺少排程必需字段

CSV 当前没有：

- 稳定 ID；
- 数值权重；
- 预计学时；
- 前置依赖；
- 适用阶段；
- 资料映射；
- 考纲年份、来源和版本。

处理方式：

- 为每条记录生成确定性 UUID/slug，保证重复导入幂等；
- 保留原始 `考纲要求` 文本，同时新增标准化等级；
- `weight`、`est_hours`、依赖边作为可编辑的系统配置，不反写原始 CSV；
- 增加 `SyllabusVersion` 和导入校验和；
- 先导入全部条目，再逐步校准权重与学时，不阻塞 P0。

### 3. “考纲知识点”和“试卷结构”需要分表

英语、政治 CSV 中包含“考试形式/试卷结构”，它们不是可学习知识点。应导入到：

```text
ExamBlueprint
ExamSection
```

用于保存总分、题型、分值和建议用时，不参与知识点 mastery 计算。

### 4. `考纲要求` 不能直接做枚举

数学一包含“会求、会用、理解掌握、理解会算、会展开”等 19 种文本。建议：

```text
requirement_raw: 原始文本
requirement_level: awareness | understanding | application | mastery
requirement_actions: JSON 数组，如 calculate/use/prove/solve
```

避免丢失原意，也便于排程器统一加权。

### 5. 前置依赖不应存 `prereq_ids[]`

建议增加关系表：

```text
KnowledgeDependency(
  predecessor_kp_id,
  successor_kp_id,
  dependency_type,
  strength
)
```

这样可做循环检测、拓扑排序和人工编辑。

### 6. 离线同步需要事件账本

仅写“last-write-win”不够。计时、打卡、拍题等操作应带：

- 客户端生成的 `event_id`；
- `device_id`；
- 客户端时间；
- 服务端接收时间；
- 幂等键；
- 同步状态和失败原因。

服务端保存 `ClientEvent` 或等价事件账本，再计算 Task 派生状态。这样重试不会重复累计耗时，也不会因两端覆盖而丢事件。

### 7. AI/图片处理必须异步

上传接口不能等待 vision、embedding、查重和分析全部完成。应增加：

```text
AIJob / MediaJob
status: queued | running | succeeded | failed
attempts
error_code
result_json
```

Android 上传成功后立即返回，审核台通过 job 状态展示处理结果。

### 8. APScheduler 与 API 多进程问题

若 FastAPI 启动多个 worker，每个 worker 都运行 APScheduler 会重复生成任务。首版二选一：

- API 单 worker + 独立 scheduler 容器；或
- scheduler 使用 PostgreSQL advisory lock 保证单实例执行。

建议独立 `worker`/`scheduler` 服务。

### 9. code_runner 必须与主服务隔离

保持独立容器，并限制：

- 禁网；
- 只读根文件系统；
- 临时工作目录；
- CPU、内存、进程数、输出大小和执行时间；
- Python 包与 C 编译器白名单；
- 不挂载数据库、图片目录或主服务密钥。

P0 不需要 code_runner，可在 AI 基础闭环稳定后接入。

### 10. 备份必须包含恢复验证

每日 `pg_dump` 还不够。需要：

- 数据库备份；
- 原图/附件备份；
- 保留策略；
- 最近备份状态；
- 定期在临时库执行恢复验证；
- 全量导出和恢复说明。

## 分阶段实施计划

## 阶段 0：工程基础与契约

目标：所有工程可编译、可启动、可迁移、可测试。

交付：

- monorepo、Git、README、环境样例；
- FastAPI、SQLAlchemy、Alembic、PostgreSQL 16 + pgvector；
- Vue 3 + Vite + TypeScript；
- Android Compose + Room + Retrofit；
- Docker Compose；
- OpenAPI 类型生成；
- 统一错误格式、日志、健康检查、单用户 token 鉴权；
- 基础 CI：后端 lint/typecheck/test、Web lint/typecheck/build、Android compile/test。

验收：

- 一条命令启动数据库、API 和 Web；
- Android debug APK 可以构建；
- 空数据库可从零迁移；
- Web 和 Android 均能调用 `/api/ping`。

## 阶段 1：P0 可用纵切片

目标：系统从第一版就能实际承担每日计划与打卡。

交付：

- 643 条考纲数据幂等导入；
- 考纲树浏览；
- 阶段、科目配比、Availability 配置；
- Material、任务模板和任务池；
- 确定性排程器；
- 今日任务、计时、打卡、顺延、请假、一键从今天重排；
- Web 日历；
- Android 今日页、Room 本地缓存、离线事件队列；
- 周时长与基本执行率。

验收：

- 可生成从当前日到 11 月底的计划；
- 同一输入重复生成结果稳定；
- 修改某日 Availability 后可重排且不破坏已完成记录；
- Android 离线打卡后恢复网络只同步一次；
- 首屏、打卡步数和响应符合设计预算。

## 阶段 2：P2 题库与 SM-2

目标：形成“拍题 → 草稿 → 审核 → 复习”的闭环。

交付：

- 图片压缩、断点/失败重试和对象存储抽象；
- 后台异步 vision 提取；
- Problem、Solution、ProblemKP；
- 查重候选；
- Web 审核台；
- 草稿题可提前进入复习；
- SM-2 卡片调度和 Android 卡片流；
- 知识点到题目的双向查询。

验收：

- 拍照后立即返回，不阻塞学习；
- AI 失败时原图和草稿不丢失；
- 一题多知识点的权重和唯一主考点约束生效；
- 复习反馈正确更新到期日；
- 超量卡片按设计自动分摊。

## 阶段 3：P3 AI、RAG 与周报

目标：先完成受约束的 AI 闭环，再接入代码验证。

交付：

- AI provider 抽象和能力探测；
- vision/chat/embedding 分开配置；
- 聊天、Memory 抽取、人工管理；
- 基于知识点 ID 的 RAG；
- 周报 SQL 事实聚合；
- plan patch Schema、校验、采纳和回滚；
- 明确的降级、重试和费用/用量记录。

验收：

- LLM 不能创建不存在的知识点 ID；
- 数字统计均来自数据库聚合；
- plan patch 超出时长或破坏依赖时不能落库；
- AI 不可用时草稿仍保存，核心计划/打卡功能不受影响。

## 阶段 4：代码验证层

目标：为可执行问题提供受隔离的验证，不把 AI 结论当事实。

交付：

- 独立 code_runner；
- Python/SymPy/NumPy/SciPy 和 C 两类执行器；
- 两段式 AI 协议；
- 超时、重试一次和“未经验证”状态；
- 代码与 stdout 在两端可审阅；
- 审计日志和资源限制。

验收：

- runner 无法访问公网、数据库和宿主敏感目录；
- 超时/编译失败不会阻塞 API；
- GPT 解法只有验证通过后才能标记 `verified=true`。

## 阶段 5：P1 看板、备份与部署

目标：补齐分析、运维与长期使用能力。

交付：

- 图谱、覆盖度、时长、执行率和真题得分；
- 周报故事页；
- 通知；
- 备份、恢复验证、全量导出；
- Caddy HTTPS、ECS 部署和 Android 更新检查；
- P4 工作区同步作为最后的可选项。

验收：

- 看板数据可追溯到事实记录；
- 新服务器可通过备份恢复；
- 部署后核心 API、Web 和 Android 完成端到端测试。

设计文档原计划 4 周完成 P0、P2、P3、P1。对于“能长期自用且数据不丢”的标准，这个周期偏激进。

- AI 模型、域名、ECS、通知和备份外部目标在对应阶段配置。
